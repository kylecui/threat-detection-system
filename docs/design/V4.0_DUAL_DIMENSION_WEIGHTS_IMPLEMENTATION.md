## 📋 执行摘要

### 问题背景

**V3.0单维度设计缺陷**:
- 仅评估攻击源网段严重性 (如: IoT设备 = 0.9低权重)
- 忽略了攻击者访问的蜜罐类型反映的攻击意图
- **关键问题**: IoT设备扫描管理蜜罐应判定为CRITICAL,但V3.0会评估为LOW

**用户场景**:
```
IoT设备 (192.168.50.10) → 管理堡垒蜜罐 (10.0.100.50)
预期: CRITICAL威胁 (低价值设备访问高敏感蜜罐 = 严重攻击意图)
V3.0结果: LOW威胁 (仅看攻击源 = 0.9低权重) ❌
```

### V4.0解决方案

**双维度评估模型**:

1. **attackSourceWeight (攻击源严重性)**: 评估"如果这台设备被入侵有多严重?"
   - 数据库服务器: 3.0 (高价值资产)
   - IoT设备: 0.9 (低价值资产)
   - 访客网络: 0.6 (隔离区域)

2. **honeypotSensitivityWeight (蜜罐敏感性)**: 评估"攻击者访问这个蜜罐反映什么意图?"
   - 管理堡垒蜜罐: 3.5 (高权限访问意图)
   - 数据库蜜罐: 3.5 (数据窃取意图)
   - 办公区蜜罐: 1.3 (一般探测行为)

3. **组合权重**: `combinedWeight = attackSourceWeight × honeypotSensitivityWeight`

**修正后的场景**:
```
IoT设备 (0.9) × 管理蜜罐 (3.5) = 3.15 → CRITICAL威胁 ✅
数据库服务器 (3.0) × 办公蜜罐 (1.3) = 3.9 → CRITICAL威胁 ✅
办公设备 (1.0) × 办公蜜罐 (1.3) = 1.3 → MEDIUM威胁 ✅
```

#### PostgreSQL查询函数

```sql
-- 函数1: 获取攻击源权重
CREATE OR REPLACE FUNCTION get_attack_source_weight(
    p_customer_id VARCHAR(50),
    p_attack_ip VARCHAR(45)
) RETURNS DECIMAL(3,2) AS $$
DECLARE
    v_weight DECIMAL(3,2);
BEGIN
    SELECT weight INTO v_weight
    FROM attack_source_weights
    WHERE customer_id = p_customer_id
      AND is_active = TRUE
      AND CAST(p_attack_ip AS inet) BETWEEN CAST(ip_range_start AS inet) AND CAST(ip_range_end AS inet)
    ORDER BY priority DESC, id DESC
    LIMIT 1;
    
    RETURN COALESCE(v_weight, 1.0);
END;
$$ LANGUAGE plpgsql;

-- 函数2: 获取蜜罐敏感性权重
CREATE OR REPLACE FUNCTION get_honeypot_sensitivity_weight(
    p_customer_id VARCHAR(50),
    p_honeypot_ip VARCHAR(45)
) RETURNS DECIMAL(3,2) AS $$
DECLARE
    v_weight DECIMAL(3,2);
BEGIN
    SELECT weight INTO v_weight
    FROM honeypot_sensitivity_weights
    WHERE customer_id = p_customer_id
      AND is_active = TRUE
      AND CAST(p_honeypot_ip AS inet) BETWEEN CAST(ip_range_start AS inet) AND CAST(ip_range_end AS inet)
    ORDER BY priority DESC, id DESC
    LIMIT 1;
    
    RETURN COALESCE(v_weight, 1.0);
END;
$$ LANGUAGE plpgsql;

-- 函数3: 获取组合权重
CREATE OR REPLACE FUNCTION get_combined_segment_weight(
    p_customer_id VARCHAR(50),
    p_attack_ip VARCHAR(45),
    p_honeypot_ip VARCHAR(45)
) RETURNS TABLE(
    attack_source_weight DECIMAL(3,2),
    honeypot_sensitivity_weight DECIMAL(3,2),
    combined_weight DECIMAL(6,4)
) AS $$
BEGIN
    RETURN QUERY
    SELECT 
        get_attack_source_weight(p_customer_id, p_attack_ip),
        get_honeypot_sensitivity_weight(p_customer_id, p_honeypot_ip),
        get_attack_source_weight(p_customer_id, p_attack_ip) * 
        get_honeypot_sensitivity_weight(p_customer_id, p_honeypot_ip);
END;
$$ LANGUAGE plpgsql;
```

#### Service Layer

**IpSegmentWeightServiceV4.java**:
```java
@Service
@RequiredArgsConstructor
@Slf4j
public class IpSegmentWeightServiceV4 {
    
    private static final double DEFAULT_ATTACK_SOURCE_WEIGHT = 1.0;
    private static final double DEFAULT_HONEYPOT_SENSITIVITY_WEIGHT = 1.0;
    
    private final AttackSourceWeightRepository attackSourceWeightRepository;
    private final HoneypotSensitivityWeightRepository honeypotSensitivityWeightRepository;
    
    /**
     * 获取攻击源权重 (第一维度)
     * 
     * @param customerId 客户ID
     * @param attackIp 攻击源IP (被诱捕的内网主机)
     * @return 权重 0.5-3.0,默认1.0
     */
    @Cacheable(value = "attackSourceWeights", key = "#customerId + ':' + #attackIp")
    public double getAttackSourceWeight(String customerId, String attackIp) {
        if (customerId == null || attackIp == null || attackIp.isBlank()) {
            return DEFAULT_ATTACK_SOURCE_WEIGHT;
        }
        
        try {
            Optional<AttackSourceWeight> weightConfig = 
                attackSourceWeightRepository.findByCustomerIdAndIpAddress(customerId, attackIp);
            
            double weight = weightConfig
                .map(AttackSourceWeight::getWeight)
                .map(BigDecimal::doubleValue)
                .orElse(DEFAULT_ATTACK_SOURCE_WEIGHT);
            
            log.debug("Attack source weight lookup: customerId={}, attackIp={}, weight={}, hasConfig={}",
                     customerId, attackIp, weight, weightConfig.isPresent());
            
            return weight;
            
        } catch (Exception e) {
            log.error("Failed to get attack source weight for customerId={}, attackIp={}: {}",
                     customerId, attackIp, e.getMessage(), e);
            return DEFAULT_ATTACK_SOURCE_WEIGHT;
        }
    }
    
    /**
     * 获取蜜罐敏感性权重 (第二维度)
     * 
     * @param customerId 客户ID
     * @param honeypotIp 蜜罐IP (诱饵IP/虚拟哨兵)
     * @return 权重 1.0-3.5,默认1.0
     */
    @Cacheable(value = "honeypotSensitivityWeights", key = "#customerId + ':' + #honeypotIp")
    public double getHoneypotSensitivityWeight(String customerId, String honeypotIp) {
        if (customerId == null || honeypotIp == null || honeypotIp.isBlank()) {
            return DEFAULT_HONEYPOT_SENSITIVITY_WEIGHT;
        }
        
        try {
            Optional<HoneypotSensitivityWeight> weightConfig = 
                honeypotSensitivityWeightRepository.findByCustomerIdAndHoneypotIp(customerId, honeypotIp);
            
            double weight = weightConfig
                .map(HoneypotSensitivityWeight::getWeight)
                .map(BigDecimal::doubleValue)
                .orElse(DEFAULT_HONEYPOT_SENSITIVITY_WEIGHT);
            
            log.debug("Honeypot sensitivity weight lookup: customerId={}, honeypotIp={}, weight={}, hasConfig={}",
                     customerId, honeypotIp, weight, weightConfig.isPresent());
            
            return weight;
            
        } catch (Exception e) {
            log.error("Failed to get honeypot sensitivity weight for customerId={}, honeypotIp={}: {}",
                     customerId, honeypotIp, e.getMessage(), e);
            return DEFAULT_HONEYPOT_SENSITIVITY_WEIGHT;
        }
    }
    
    /**
     * 获取组合权重 (双维度乘积)
     * 
     * @param customerId 客户ID
     * @param attackIp 攻击源IP
     * @param honeypotIp 蜜罐IP
     * @return 组合权重 = attackSourceWeight × honeypotSensitivityWeight
     */
    public double getCombinedSegmentWeight(String customerId, String attackIp, String honeypotIp) {
        double attackSourceWeight = getAttackSourceWeight(customerId, attackIp);
        double honeypotSensitivityWeight = getHoneypotSensitivityWeight(customerId, honeypotIp);
        double combinedWeight = attackSourceWeight * honeypotSensitivityWeight;
        
        log.info("Combined segment weight: customerId={}, attackIp={}, honeypotIp={}, " +
                "attackSourceWeight={}, honeypotSensitivityWeight={}, combinedWeight={}",
                customerId, attackIp, honeypotIp,
                attackSourceWeight, honeypotSensitivityWeight, combinedWeight);
        
        return combinedWeight;
    }
    
    // ... Optional<> getters, count methods ...
}
```

#### 当前限制

**未使用 `honeypotSensitivityWeight` 的原因**:
- `AggregatedAttackData` DTO缺少 `responseIp` 或 `mostAccessedHoneypotIp` 字段

**影响**:
- ✅ 仍然比V3.0有显著改进 (考虑了攻击源严重性)
- ⚠️ 未完全实现V4.0设计 (缺少蜜罐敏感性维度)
- ⚠️ IoT → 管理蜜罐场景会被评为 `0.9 × (默认) = 低威胁`,而非期望的 `0.9 × 3.5 = 3.15 → CRITICAL`

```java
public double calculateThreatScore(AggregatedAttackData data) {
    // ... 现有逻辑 ...
    
    // 添加蜜罐敏感性权重
    double honeypotSensitivityWeight = 1.0;
    if (data.getMostAccessedHoneypotIp() != null) {
        honeypotSensitivityWeight = ipSegmentWeightServiceV4
            .getHoneypotSensitivityWeight(data.getCustomerId(), 
                                         data.getMostAccessedHoneypotIp());
    }
    
    // 更新最终评分
    double finalScore = baseScore * timeWeight * ipWeight * portWeight 
                      * deviceWeight * attackSourceWeight * honeypotSensitivityWeight;
    
    return finalScore;
}
```

**方案B: 在Flink层预计算双维度权重**

1. **在Flink聚合函数中调用PostgreSQL**:
   - 对每个攻击事件,查询 `get_combined_segment_weight()`
   - 在窗口聚合时计算平均组合权重或最大组合权重
   - 直接存储到 `AggregatedAttackData.precomputedSegmentWeight`

2. **优点**: 
   - ThreatScoreCalculator无需修改
   - 权重计算提前到聚合层,更精确

3. **缺点**:
   - Flink需要连接PostgreSQL (增加复杂性)
   - 缓存策略需要重新设计

**推荐**: 方案A (简单,清晰,易于维护)

**缓存后端**: Redis (通过Spring Boot缓存配置)

**缓存命中率预期**:
- 攻击源IP: 高命中率 (内网IP数量有限,重复攻击常见)
- 蜜罐IP: 极高命中率 (蜜罐IP固定配置)

**预期性能提升**:
- 无缓存: ~50-100ms SQL查询 × 1000 events/s = 50-100% CPU开销
- 有缓存: ~0.1ms缓存查询 × 1000 events/s = 0.1% CPU开销

```sql
CREATE INDEX idx_asw_customer_active ON attack_source_weights(customer_id, is_active);
CREATE INDEX idx_asw_ip_range ON attack_source_weights USING gist 
    (int8range(inet_to_bigint(ip_range_start::inet), inet_to_bigint(ip_range_end::inet)));

-- honeypot_sensitivity_weights表
CREATE INDEX idx_hsw_customer_active ON honeypot_sensitivity_weights(customer_id, is_active);
CREATE INDEX idx_hsw_ip_range ON honeypot_sensitivity_weights USING gist 
    (int8range(inet_to_bigint(ip_range_start::inet), inet_to_bigint(ip_range_end::inet)));
```

**查询性能**:
- INET BETWEEN查询: ~1-5ms (有索引)
- 默认值返回: ~0.1ms (无数据库访问)

## 🔍 故障排查

### 问题1: 权重始终返回1.0

**症状**:
```
V4.0 attack source weight applied: customerId=customer-001, attackIp=192.168.50.10, weight=1.0
```

**可能原因**:
1. 数据库配置未加载 (检查SQL脚本执行)
2. IP地址不在配置范围内
3. `is_active = FALSE`
4. `customer_id` 不匹配

**排查步骤**:
```sql
-- 1. 检查配置是否存在
SELECT * FROM attack_source_weights 
WHERE customer_id = 'customer-001' 
  AND '192.168.50.10'::inet BETWEEN ip_range_start::inet AND ip_range_end::inet;

-- 2. 检查INET转换
SELECT CAST('192.168.50.10' AS inet);

-- 3. 手动调用函数
SELECT get_attack_source_weight('customer-001', '192.168.50.10');

-- 4. 检查is_active状态
SELECT segment_name, is_active FROM attack_source_weights 
WHERE customer_id = 'customer-001';
```

### 问题2: 性能下降

**症状**: 威胁评分延迟增加

**可能原因**:
1. 缓存未启用
2. 数据库索引缺失
3. 高并发下的数据库连接池耗尽

**排查步骤**:
```bash
# 1. 检查缓存命中率
docker-compose exec redis redis-cli INFO stats | grep keyspace

# 2. 检查数据库索引
docker-compose exec postgres psql -U postgres -d threat_detection -c "
SELECT schemaname, tablename, indexname 
FROM pg_indexes 
WHERE tablename IN ('attack_source_weights', 'honeypot_sensitivity_weights');"

# 3. 检查数据库连接
docker-compose exec postgres psql -U postgres -c "
SELECT count(*), state FROM pg_stat_activity 
WHERE datname = 'threat_detection' 
GROUP BY state;"
```
