## 1. 系统核心概念

### 这是什么系统？

本系统是一套**内网蜜罐/欺骗防御威胁检测平台**。它不是传统的边界防火墙或 IDS，而是在内网中部署虚拟哨兵（诱饵 IP），通过诱导已被攻破的内部主机暴露其横向移动行为来发现威胁。

### 蜜罐机制——你必须理解的三个字段

在整个系统中，所有数据围绕三个核心字段展开：

| 字段 | 含义 | 安全意义 |
|------|------|----------|
| `attack_ip` / `attack_mac` | **被诱捕的内网主机**的 IP 和 MAC 地址 | 这台主机已被攻破或感染，正在内网中横向移动 |
| `response_ip` | **诱饵 IP**（虚拟哨兵，物理上不存在的地址） | 任何访问该 IP 的行为都是恶意的——因为正常业务不可能访问不存在的地址 |
| `response_port` | **攻击者尝试访问的端口** | 暴露攻击者的真实意图（如 3389 = 远程控制，445 = 横向移动，3306 = 数据窃取） |

> **关键认知**：由于 `response_ip` 是不存在的虚拟哨兵，所有对它的访问都是**确认的恶意行为**。这意味着系统的误报率极低，你看到的每一条记录都值得关注。

### 数据是怎么来的？

```
终端蜜罐设备 (Sentry) 部署在各个内网网段
        ↓ 监控二层网络，识别未使用 IP 作为诱饵
        ↓ 主动响应 ARP/ICMP 诱导攻击者
        ↓ 记录攻击者对诱饵的端口访问
        ↓
数据摄取服务 (syslog/MQTT) → Kafka
        ↓
Flink 实时流处理 (30s/5min/15min 三级窗口)
        ↓
威胁评分 → 威胁情报增强 → ML 异常检测
        ↓
告警生成 → 多通道通知 (邮件/SMS/Slack/Webhook/Teams)
```

## 2. 仪表盘（概览页）

概览页是你登录后的第一个页面，提供当前威胁态势的全局视图。

### 2.1 顶部统计卡片

页面顶部有 **4 张统计卡片**，分别展示当前客户范围内的威胁评估总数和各等级计数：

| 卡片 | 含义 | 运维关注点 |
|------|------|------------|
| **威胁总数** | 所有威胁评估记录的总量 | 反映整体攻击活跃度。如果短时间内总量激增，说明可能有大规模扫描或蠕虫传播事件 |
| **严重威胁** (CRITICAL) | 评分 ≥80 的威胁数量（红色） | **必须立即处理**。通常意味着攻击者已在多个网段间快速横向移动，攻击意图明确 |
| **高危威胁** (HIGH) | 评分 60-79 的威胁数量（橙色） | 需要在当日内调查处理。可能是定向攻击的早期阶段 |
| **中危威胁** (MEDIUM) | 评分 40-59 的威胁数量（金色） | 需要在本周内关注。可能是自动化扫描工具或受感染主机的探测行为 |

> **操作建议**：每天上班第一件事，看 CRITICAL 和 HIGH 数字。如果比昨天明显增多，立即进入威胁列表排查。

### 2.2 威胁趋势图

位于统计卡片下方的**折线图**，展示威胁数量随时间的变化趋势。

- **时间范围**：支持 24 小时 / 7 天 / 30 天 切换
- **X 轴**：时间
- **Y 轴**：威胁评估数量

**怎么看这个图？**

- **正常态势**：曲线平稳，偶有小幅波动
- **值得注意**：某个时段出现明显尖峰——可能是集中扫描事件
- **需要警惕**：趋势持续上升——可能有新的攻击源持续活动
- **深夜尖峰**：凌晨 0:00-6:00 出现高峰特别值得关注，这个时段的异常行为更可能是 APT 活动（系统会对此给出更高的时间权重）

### 2.3 威胁等级分布（饼图）

展示 CRITICAL / HIGH / MEDIUM / LOW / INFO 五个等级各自占比。

**怎么看这个图？**

- **健康分布**：大部分为 INFO 和 LOW，少量 MEDIUM，极少 HIGH/CRITICAL
- **异常分布**：HIGH 和 CRITICAL 占比超过 20%——说明网络中有严重的横向移动活动
- **全部为 INFO**：网络相对安全，或者攻击者活动非常轻微（单次探测）

### 2.4 端口分布（柱状图）

展示 **Top 15 被攻击端口**的分布情况，按攻击次数排序。

这个图直接揭示攻击者的**攻击意图**：

| 常见端口 | 攻击意图 | 运维动作 |
|----------|----------|----------|
| 445 (SMB) | 横向移动 / 勒索软件传播 | 检查是否有 WannaCry 类蠕虫活动 |
| 3389 (RDP) | 远程控制 | 检查是否有暴力破解 RDP 的行为 |
| 3306 (MySQL) | 数据窃取 | 检查数据库服务器安全配置 |
| 22 (SSH) | 远程登录 | 检查 SSH 密钥和密码策略 |
| 80/443 (HTTP/S) | Web 攻击 | 检查 Web 应用漏洞 |
| 139 (NetBIOS) | 网络枚举 | 攻击者在收集网络拓扑信息 |
| 1433 (MSSQL) | 数据窃取 | 检查 SQL Server 安全配置 |
| 5432 (PostgreSQL) | 数据窃取 | 检查数据库访问控制 |
| 6379 (Redis) | 未授权访问 | Redis 经常暴露在内网无密码状态 |

> **操作建议**：如果 445 端口突然出现在榜首，立即检查是否有勒索软件传播。如果出现大量非常见端口（如高端口），可能是攻击者在尝试绕过安全策略。

### 2.5 Top 攻击者表格

展示攻击活动最活跃的内网主机列表：

| 列 | 含义 |
|----|------|
| 排名 | 按总攻击次数排序 |
| 攻击者 MAC | 被攻破的内网主机 MAC 地址——这是**需要你去隔离和处理的设备** |
| 攻击者 IP | 被攻破的内网主机 IP 地址 |
| 总攻击次数 | 该主机对诱饵的总探测次数 |
| 最高威胁分数 | 该主机产生的最高单次评分 |
| 最高威胁等级 | 该主机的最严重等级 |

> **操作建议**：表格中排名靠前的主机是**最紧急需要处理的**。它们已经确认被攻破，正在积极进行横向移动。应立即隔离这些主机并进行取证分析。

### 2.6 最近威胁表格

## 3. 威胁列表

威胁列表是你日常运维中使用最多的页面，展示所有威胁评估记录的详细信息。

### 3.1 筛选功能

| 筛选器 | 用法 |
|--------|------|
| **威胁等级** | 下拉选择：ALL / CRITICAL / HIGH / MEDIUM / LOW / INFO。建议日常使用时筛选 CRITICAL 和 HIGH |
| **时间范围** | 日期选择器，可选择起止日期。调查特定时间段的事件时使用 |
| **攻击者 MAC** | 搜索框，支持模糊匹配。当你已经知道某台主机可疑时，直接搜索其 MAC 地址 |

### 3.2 表格各列含义

| 列名 | 含义 | 关注要点 |
|------|------|----------|
| **ID** | 记录唯一标识 | 用于内部引用和 API 查询 |
| **评估时间** | 该威胁评估产生的时间 | 多条记录聚集在同一时段意味着攻击正在进行 |
| **攻击者 MAC** | 被攻破的内网主机 MAC | 相同 MAC 反复出现说明该主机持续活动，尚未被隔离 |
| **攻击者 IP** | 被攻破的内网主机 IP | 与 MAC 配合定位具体设备 |
| **威胁等级** | CRITICAL / HIGH / MEDIUM / LOW / INFO | 颜色编码——红色 > 橙色 > 金色 > 蓝色 > 灰色 |
| **威胁评分** | 综合评分（已归一化到 1-100） | 数值越高，横向移动越广泛、攻击意图越明确 |
| **攻击次数** | 在评估窗口内的总探测次数 | 高攻击次数 = 快速扫描或蠕虫式传播 |
| **唯一 IP 数** | 访问了多少个不同的诱饵 IP | 高唯一 IP 数 = 大范围横向移动 |
| **唯一端口数** | 尝试了多少种不同的端口 | 高唯一端口数 = 攻击意图多样化（同时尝试多种攻击） |
| **唯一设备数** | 被多少个不同的蜜罐设备检测到 | 高唯一设备数 = 攻击已跨越多个网段 |

### 3.3 详情抽屉

点击任意一行，右侧会弹出详情抽屉，显示完整的评估信息：

| 字段 | 含义 |
|------|------|
| **端口列表** | 攻击者尝试访问的所有端口——每个端口直接映射攻击者的一个攻击意图 |
| **端口风险评分** | 基于端口危险程度的综合评分 |
| **检测层级** (Detection Tier) | 触发该评估的时间窗口层级：Tier 1 (30s) / Tier 2 (5min) / Tier 3 (15min)，详见 [第 6 节](#6-三级时间窗口机制) |
| **缓解建议** | 系统自动生成的处置建议 |

### 3.4 批量操作

- **批量删除**：勾选多条记录后批量删除（用于清理已处理的记录）
- **导出 CSV / JSON**：导出当前页数据用于离线分析或报告

## 4. 告警中心

告警中心管理由威胁评估自动触发的告警，支持完整的告警处置工作流。

### 4.1 告警统计卡片

| 卡片 | 含义 |
|------|------|
| **告警总数** | 所有告警记录的总量 |
| **已解决** | 已处置完毕的告警数量（绿色） |
| **已升级** | 被升级到更高处理层级的告警数量（红色） |
| **平均解决时间** | 从告警生成到解决的平均耗时 |

### 4.2 告警状态流转

一条告警会经历以下状态：

```
NEW (新建)
  → DEDUPLICATED (已去重——相同攻击者的重复告警被合并)
  → ENRICHED (已增强——TIRE 威胁情报已补充上下文)
  → NOTIFIED (已通知——邮件/SMS/Slack 已发送)
  → ESCALATED (已升级——需要更高级别人员介入)
  → RESOLVED (已解决——处置完毕)
  → ARCHIVED (已归档)
```

### 4.3 告警处置操作

| 操作 | 何时使用 | 说明 |
|------|----------|------|
| **解决** (Resolve) | 确认威胁已处理完毕 | 填写解决方案描述和处理人 |
| **分配** (Assign) | 需要指定责任人跟进 | 填写负责人姓名或工号 |
| **升级** (Escalate) | 自己无法处理，需要上级介入 | 填写升级原因。升级后告警显示升级级别 (L1, L2...) |
| **取消升级** | 误升级或情况已变化 | 直接取消，无需填写原因 |

### 4.4 告警字段说明

| 列名 | 含义 |
|------|------|
| **标题** | 告警概述（如"CRITICAL: MAC xx:xx:xx 高频横向移动"） |
| **严重程度** | CRITICAL / HIGH / MEDIUM / LOW / INFO |
| **状态** | 当前处置状态（见上方状态流转） |
| **攻击者 MAC** | 触发告警的内网失陷主机 |
| **威胁评分** | 触发告警时的威胁评分 |
| **升级级别** | L0 = 未升级，L1+ = 已升级（数字越大越紧急） |
| **负责人** | 当前指派的处理人员 |
| **创建时间** | 告警生成时间 |

## 5. 威胁评分算法

理解评分算法可以帮助你判断为什么某条威胁记录得到了高分或低分。

### 5.1 评分公式

```
威胁评分 = (攻击次数 × 唯一IP数 × 唯一端口数)
         × 时间权重 × IP权重 × 端口权重 × 设备权重
         × 网段权重 × 攻击速率权重 × ML权重
```

用大白话说就是：**基础分 = 攻击的广度和频率，然后被各种环境因子放大或缩小。**

### 5.2 基础分的含义

| 因子 | 安全含义 |
|------|----------|
| **攻击次数** (attackCount) | 频率越高 → 攻击越积极（蠕虫式快速传播 vs 低慢APT探测） |
| **唯一 IP 数** (uniqueIps) | 访问的诱饵越多 → 横向移动范围越大（在扫描多少个"房间"） |
| **唯一端口数** (uniquePorts) | 尝试的端口越多 → 攻击意图越多样（在尝试多少种"钥匙"） |

三个因子相乘，所以**同时具有高频率、广范围、多意图的攻击会得到极高的基础分**。

### 5.3 威胁等级映射

评分归一化到 1-100 后，映射为五个等级：

| 等级 | 分数范围 | 颜色 | 含义 | 建议响应时间 |
|------|----------|------|------|-------------|
| **CRITICAL** | ≥ 80 | 🔴 红色 | 严重威胁——大规模横向移动，攻击意图明确且多样 | **立即** (< 15 分钟) |
| **HIGH** | 60-79 | 🟠 橙色 | 高危威胁——显著的横向移动行为 | **当日** (< 4 小时) |
| **MEDIUM** | 40-59 | 🟡 金色 | 中危威胁——有一定的探测行为 | **本周** (< 24 小时) |
| **LOW** | 20-39 | 🔵 蓝色 | 低危威胁——轻微的探测活动 | **监控** (持续观察) |
| **INFO** | 1-19 | ⚪ 灰色 | 信息级别——单次或极少量的探测 | **记录** (存档备查) |

## 6. 三级时间窗口机制

系统使用 Apache Flink 进行实时流处理，采用三级时间窗口捕获不同速度的攻击模式：

### Tier 1 — 30 秒窗口（勒索软件检测）

- **检测目标**：快速传播的蠕虫和勒索软件（如 WannaCry）
- **特征**：在 30 秒内对大量 IP 的 445 端口发起连接
- **典型场景**：一台被感染的主机在几秒内尝试连接网段内所有可达 IP 的 SMB 端口
- **如果你看到 Tier 1 告警**：⚠️ **这是最紧急的情况**，可能有勒索软件正在传播。立即隔离相关主机，通知所有网段管理员。

### Tier 2 — 5 分钟窗口（主要威胁检测）

- **检测目标**：大多数常规攻击行为
- **特征**：在 5 分钟内持续扫描多个诱饵、尝试多个端口
- **典型场景**：攻击者使用扫描工具对一个子网进行系统性探测
- **如果你看到 Tier 2 告警**：这是最常见的告警类型。评估攻击范围后决定是否需要立即隔离。

### Tier 3 — 15 分钟窗口（APT 检测）

- **检测目标**：高级持续性威胁 (APT) 的低慢横向移动
- **特征**：在 15 分钟内缓慢地、有策略地探测少量目标
- **典型场景**：攻击者每隔几分钟才探测一个新目标，试图规避检测
- **如果你看到 Tier 3 告警**：虽然评分可能不高，但这种行为模式本身就高度可疑。APT 攻击者通常更有耐心，更危险。结合威胁情报数据综合判断。

## 7. 评分权重因子详解

每个权重因子都有其安全含义。理解它们可以帮助你分析为什么某条记录的评分特别高。

### 7.1 时间权重 (timeWeight): 0.8 - 1.2

根据攻击发生的时间段调整：

| 时间段 | 权重 | 原因 |
|--------|------|------|
| 0:00-6:00 | **1.2** | 深夜——正常业务已停止，此时的网络活动更可能是恶意的（APT活动高发期） |
| 6:00-9:00 | 1.1 | 早晨——业务尚未完全开始 |
| 9:00-17:00 | 1.0 | 工作时间——基准值 |
| 17:00-21:00 | 0.9 | 傍晚——正常下班时间 |
| 21:00-24:00 | 0.8 | 夜间——可能有值班人员的正常活动 |

> **怎么用**：如果一条 HIGH 级别的威胁发生在凌晨 3 点，它的权重被放大了 1.2 倍——系统认为这比白天同等行为更可疑。你也应该更重视它。

### 7.2 IP 权重 (ipWeight): 1.0 - 2.0

根据攻击者访问的诱饵 IP 数量（横向移动范围）调整：

| 唯一 IP 数 | 权重 | 含义 |
|-----------|------|------|
| 1 | 1.0 | 仅访问了一个诱饵——可能是偶发行为 |
| 2-3 | 1.3 | 小范围扫描——开始有横向移动迹象 |
| 4-5 | 1.5 | 中等范围——有系统性地探测网段 |
| 6-10 | 1.7 | 广泛扫描——确认在大范围横向移动 |
| 10+ | **2.0** | 大规模扫描——已触及网段内大量主机 |

### 7.3 端口权重 (portWeight): 1.0 - 2.0

根据攻击者尝试的端口种类数量（攻击意图多样性）调整：

| 唯一端口数 | 权重 | 含义 |
|-----------|------|------|
| 1 | 1.0 | 单一端口——目标明确（如只尝试 SMB） |
| 2-3 | 1.2 | 少量端口——有限的攻击意图 |
| 4-5 | 1.4 | 中等——在试探多种攻击路径 |
| 6-10 | 1.6 | 广泛——使用扫描工具进行系统性探测 |
| 11-20 | 1.8 | 大范围——全面的服务探测 |
| 20+ | **2.0** | 全端口扫描——可能使用了 Nmap 全端口扫描 |

### 7.4 设备权重 (deviceWeight): 1.0 - 1.5

| 唯一设备数 | 权重 | 含义 |
|-----------|------|------|
| 1 | 1.0 | 仅被一个蜜罐设备检测到——攻击限于单一网段 |
| 2+ | **1.5** | 被多个蜜罐设备检测到——**攻击已跨越网段边界** |

> **重要**：当 deviceWeight = 1.5 时，意味着攻击者已经在不同的网段间移动，这是一个**严重的网络安全事件**的信号。

### 7.5 攻击源权重 (attackSourceWeight): 0.5 - 3.0

根据被攻破设备的类型调整：

| 设备类型 | 权重 | 原因 |
|----------|------|------|
| 普通办公终端 | 1.0 | 基准值 |
| IoT 设备 | 0.9 | IoT 设备可能产生一些噪音流量 |
| **数据库服务器** | **3.0** | 数据库服务器被攻破意味着数据泄露风险极高 |

### 7.6 蜜罐敏感度权重 (honeypotSensitivityWeight): 1.0 - 3.5

根据被攻击的诱饵类型调整：

| 蜜罐类型 | 权重 | 原因 |
|----------|------|------|
| 普通办公诱饵 | 1.3 | 模拟普通办公主机 |
| **管理网络诱饵** | **3.5** | 攻击者在尝试访问管理网络——意味着深度渗透 |

### 7.7 攻击速率权重 (attackRateWeight): 0.5 - 2.5

根据每秒攻击频率调整：

| 攻击速率 | 权重 | 含义 |
|----------|------|------|
| < 0.01/秒 | 0.5 | 极慢扫描——低慢 APT 行为，降低权重以减少噪音 |
| 0.01-0.1/秒 | 0.8 | 慢速扫描 |
| 0.1-1/秒 | 1.0 | 中等速率——基准值 |
| 1-10/秒 | 1.5 | 快速扫描——自动化工具 |
| > 10/秒 | **2.5** | DDoS 级别爆发——极高频率攻击 |

### 7.8 ML 权重 (mlWeight): 0.5 - 3.0

由机器学习异常检测模型自动计算的倍率：

| 值范围 | 含义 |
|--------|------|
| 0.5 | ML 模型认为此行为与正常模式高度相似——降低评分 |
| 1.0 | 中性——ML 无法判断 |
| **3.0** | ML 模型检测到高度异常的行为模式——大幅提升评分 |

> **注意**：mlWeight 是**建议性**的，可通过功能开关控制是否启用。当 ML 服务不可用时，默认值为 1.0（不影响评分）。

## 8. ML 异常检测

ML 检测页面展示机器学习服务的运行状态和模型信息。

### 8.1 服务状态卡片

| 指标 | 含义 |
|------|------|
| **服务状态** | ML 检测服务是否正常运行 (ok / abnormal) |
| **模型加载** | 当前是否有可用的推理模型 (yes / no) |
| **Kafka 连接** | ML 服务与消息队列的连接状态 |

### 8.2 模型架构

系统使用**双模型架构**进行异常检测：

- **自编码器 (Autoencoder)**：学习正常流量的特征分布，通过重建误差识别异常
- **BiGRU (双向门控循环单元)**：捕捉攻击行为的时序模式，识别序列级别的异常

每个时间窗口层级（Tier 1/2/3）都有独立的模型，分别优化了对不同攻击速度的检测能力。

### 8.3 模型训练

| 字段 | 含义 |
|------|------|
| **训练数据就绪度** | 各层级（Tier）的样本数量是否达到训练所需的最低要求 |
| **训练状态** | 当前是否有正在进行的训练任务 |
| **上次训练结果** | 最近一次训练是否成功，耗时多少 |

### 8.4 运行时统计

| 指标 | 含义 |
|------|------|
| **序列缓冲区** (Sequence Buffer) | 用于时序分析的事件缓存——总键数和窗口数 |
| **漂移检测** (Drift Detection) | 监测数据分布是否偏离训练数据——如果检测到漂移，模型可能需要重新训练 |
| **影子评分** (Shadow Scoring) | 新模型与当前模型并行运行的对比——用于安全地验证新模型的效果 |

> **运维关注点**：如果漂移检测频繁触发，说明网络环境已经变化（如新增设备、网段调整），建议重新训练模型。

### 9.1 统计卡片

| **情报源数量** (Feed Count) | 已接入的外部情报源数量 |

### 9.2 IP 信誉查询

> **使用场景**：当你在威胁列表中发现一个可疑的 attack_ip 时，可以到威胁情报页面查询该 IP 是否在外部威胁情报库中有记录。如果命中，说明这台主机可能已被用作已知攻击活动的基础设施。

### 9.3 指标管理

指标 (IOC) 表格展示所有威胁指标的详细信息：

| 列名 | 含义 |

### 9.4 情报源管理

## 10. 安全运维操作流程

### 10.1 日常巡检流程

```
1. 登录系统 → 查看概览页统计卡片
   ├── CRITICAL > 0 → 立即进入威胁列表排查（见 10.2）
   ├── HIGH 明显增多 → 检查趋势图确认时间范围，进入威胁列表
   └── 一切正常 → 查看趋势图确认近 24h 无异常尖峰

2. 查看端口分布
   ├── 445/139 突增 → 可能有勒索软件传播，紧急排查
   ├── 出现罕见端口 → 可能有新型攻击工具
   └── 分布正常 → 继续

3. 查看 Top 攻击者
   ├── 新 MAC 出现 → 有新的主机被攻破
   ├── 旧 MAC 仍在 → 之前的处置可能未生效
   └── 列表清空 → 网络态势良好

4. 检查告警中心
   ├── 有未处理告警 → 分配/处理
   └── 全部已解决 → 归档
```

### 10.2 威胁调查流程

当你发现一条需要调查的威胁记录：

```
1. 在威胁列表中点击该记录，查看详情抽屉
   → 记录 attack_mac, attack_ip, 端口列表, 检测层级

2. 分析攻击意图
   → 端口列表中有什么？
   → 445/139 = SMB横向移动
   → 3389 = RDP远程控制
   → 3306/1433/5432 = 数据库攻击
   → 高端口 = 可能是自定义后门

3. 评估攻击范围
   → 唯一 IP 数高 = 大范围扫描，需要检查其他网段
   → 唯一设备数 > 1 = 已跨网段，影响范围更大
   → 检测层级 = Tier 1 → 勒索软件速度，最紧急

4. 查询威胁情报
   → 到威胁情报页面查询 attack_ip
   → 如果命中已知 IOC → 确认为已知攻击团伙/工具
   → 如果未命中 → 可能是新型攻击

5. 采取行动
   → 立即隔离 attack_mac 对应的主机（断网/VLAN隔离）
   → 通知资产管理员确认该主机的业务用途
   → 对该主机进行取证分析（恶意进程、后门文件、异常连接）
   → 检查同网段其他主机是否也有被攻破的迹象

6. 在告警中心记录处置结果
   → 填写解决方案和处理人
   → 标记告警为已解决
```

### 10.3 应急响应清单

当出现 **CRITICAL** 级别威胁时：

| 步骤 | 动作 | 时限 |
|------|------|------|
| 1 | 确认 attack_mac 对应主机，**立即网络隔离** | < 5 分钟 |
| 2 | 检查检测层级：Tier 1 = 全网通报勒索软件风险 | < 10 分钟 |
| 3 | 在威胁列表中搜索同一 MAC 的历史记录，确定首次出现时间 | < 15 分钟 |
| 4 | 查看端口分布，判断攻击意图和可能的横向移动路径 | < 15 分钟 |
| 5 | 查询威胁情报，确认是否为已知攻击活动 | < 20 分钟 |
| 6 | 检查 uniqueDevices > 1 的情况，确认是否有跨网段传播 | < 20 分钟 |
| 7 | 通知相关网段管理员，启动全面排查 | < 30 分钟 |
| 8 | 对被隔离主机进行取证分析 | < 2 小时 |
| 9 | 在告警中心记录完整的事件处置报告 | < 4 小时 |

### 10.4 关键判断经验

| 你看到的现象 | 可能的含义 | 建议动作 |
|--------------|-----------|----------|
| 同一 MAC 反复出现，每次分数都很高 | 主机持续被控制，攻击者仍在活动 | 确认之前的隔离措施是否生效 |
| 多个不同 MAC 攻击相同端口 | 可能是蠕虫自动传播 | 检查该端口对应的漏洞，全网排查 |
| 深夜出现 Tier 1 告警 | 高度可疑的自动化攻击 | 立即响应，不要等到白天 |
| 唯一 IP 数 > 10 但攻击次数不高 | 系统性的网络侦察 | 攻击者在收集信息，可能是 APT 前期 |
| ML 权重 > 2.0 | ML 模型检测到异常模式 | 该行为与历史正常模式显著不同，额外关注 |
| TIRE 情报命中已知 IOC | 与已知威胁活动关联 | 参考对应威胁报告中的 TTPs 和防御建议 |
