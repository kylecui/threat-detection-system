## 🔴 Critical 困难 (项目成败关键)

### 1. kSwitch eBPF 蜜罐方案 ⭐⭐⭐⭐⭐

**困难描述**:
- **外部框架依赖**: kSwitch 是您研究的独立项目，不是成熟开源框架
- **技术栈跨度**: 需要深入掌握 eBPF/XDP、内核网络栈、BPF Maps
- **开发复杂度**: 需要编写内核态 C 代码 + 用户态 Java 代码，调试困难
- **性能风险**: XDP 程序错误可能导致网络瘫痪

**具体挑战**:

| 挑战点 | 难度 | 说明 |
|--------|------|------|
| **eBPF Verifier 限制** | 🔴 极高 | 指令数限制 (4096条)，循环展开，无法使用标准库 |
| **协议模拟复杂度** | 🔴 极高 | SMB/RDP/SSH 协议栈庞大，eBPF 无法完整实现 |
| **内核态调试** | 🔴 极高 | 无法使用 gdb，只能用 bpf_printk，调试效率低 |
| **BPF Maps 性能** | 🟡 高 | 大规模查询 (1024个诱饵IP) 可能成为瓶颈 |
| **Ring Buffer 丢数据** | 🟡 高 | 高并发下 Ring Buffer 可能满，导致事件丢失 |

**技术依赖**:
```
必需技能:
- eBPF/XDP 专家 (至少1人，2年以上经验)
- Linux 内核网络栈深度理解
- C 语言底层编程
- libbpf/bpftool 工具链

必需基础设施:
- Linux Kernel >= 5.10 (XDP 完整支持)
- LLVM/Clang >= 10.0 (eBPF 编译)
- libbpf >= 0.4 (用户态库)

外部依赖:
- kSwitch 项目需要完整编译和测试
- 可能需要修改 kSwitch 源码以适配蜜罐场景
```

**实施风险**:
```
高风险场景:
1. eBPF 程序 bug 导致网络中断 (生产环境灾难性影响)
2. 协议模拟不完整，高级攻击者识别为蜜罐
3. 性能瓶颈导致丢包，影响正常业务流量
4. kSwitch 与现有网络架构不兼容

应急预案:
- 必须在隔离测试环境充分验证 (至少2个月)
- 准备降级方案: 回退到无交互蜜罐 (L0)
- 限制蜜罐部署范围: 先在非关键子网试点
```

**建议**:
```
✅ 可行性验证 (Week 1-4):
   - 编译 kSwitch，验证基本功能
   - 实现最小化 TCP SYN-ACK 响应
   - 性能测试: 确保 < 5% 性能损失

⚠️ 降级方案:
   - 如果 eBPF 方案失败，改用用户态蜜罐:
     - Dionaea (中交互，容器化)
     - Cowrie (SSH/Telnet)
     - 直接跳过 L1，只实施 L2/L3

🔄 替代方案:
   - 考虑成熟的开源蜜罐框架:
     - T-Pot (集成多个蜜罐)
     - Modern Honey Network (MHN)
   - 优点: 成熟稳定，社区支持
   - 缺点: 无法与 kSwitch 的可重配置特性结合
```

### 2. ML 模型训练数据标注 ⭐⭐⭐⭐⭐

**困难描述**:
- **人工成本巨大**: 需要 2000+ 样本，每条标注 2-5 分钟，合计 **67-167 小时**
- **标注质量要求高**: 需要安全分析师专业判断，成本高 (按 $100/小时计)
- **数据不平衡**: 真实威胁/误报比例可能是 70:30，需要特殊处理

**具体挑战**:

| 挑战点 | 难度 | 说明 |
|--------|------|------|
| **标注成本** | 🔴 极高 | 67-167小时 × $100/小时 = **$6,700 - $16,700** |
| **标注质量** | 🔴 极高 | 需要双盲标注，Cohen's Kappa > 0.75 |
| **数据收集** | 🟡 高 | 需要 6 个月历史数据，可能没有足够的"已知误报"样本 |
| **类别不平衡** | 🟡 高 | 真实威胁可能只有 30%，需要 SMOTE 等过采样技术 |

**依赖条件**:
```
人力资源:
- 安全分析师 >= 2人 (避免单点偏见)
- 标注时间: 每人 40-80 小时
- 质量审核: 第三方专家抽检 10%

数据资源:
- 历史告警数据 >= 6 个月
- 已知误报案例 >= 600 条
- 已知真实威胁案例 >= 1400 条

技术资源:
- 标注工具开发 (Flask Web界面)
- 标注进度跟踪
- 双盲标注一致性检查
```

**实施风险**:
```
高风险场景:
1. 标注数据不足 (< 2000条) → 模型欠拟合
2. 标注质量差 (一致性 < 0.75) → 模型学习错误模式
3. 类别不平衡 (真实威胁 < 30%) → 模型偏向误报
4. 标注时间超期 (> 2个月) → 项目延期

应急预案:
- 降低标注要求: 从 2000 条降到 1000 条 (牺牲模型准确性)
- 使用半监督学习: 只标注 500 条，其余用伪标签
- 外包标注: 寻找安全服务商协助 (成本更高)
```

**建议**:
```
✅ 分阶段标注:
   Week 17: 标注 500 条 → 训练初步模型 → 评估效果
   Week 18: 根据效果决定是否继续标注

✅ 主动学习 (Active Learning):
   - 模型预测不确定的样本优先标注
   - 减少标注量到 1200-1500 条

⚠️ 降级方案:
   - 如果 ML 效果不佳 (准确率 < 90%)
   - 改用纯规则过滤:
     - Windows Update 特征 → 白名单
     - 已知扫描工具 (nmap, masscan) → 白名单
     - 业务系统 IP → 白名单
```

### 3. APT 状态机真实数据验证 ⭐⭐⭐⭐

**困难描述**:
- **缺乏真实 APT 样本**: 文档中的 APT 检测都是理论模型，没有真实数据验证
- **攻击链复杂度**: APT 攻击持续数周到数月，很难在测试环境模拟
- **状态转换判定**: 如何准确判断从"侦察"到"利用"的转换？阈值如何设定？

**具体挑战**:

| 挑战点 | 难度 | 说明 |
|--------|------|------|
| **真实 APT 数据获取** | 🔴 极高 | 公开 APT 数据集稀少，内部可能没有 APT 案例 |
| **状态转换阈值** | 🔴 极高 | "侦察 → 投递" 如何判定？仅凭端口访问次数？ |
| **时间窗口设计** | 🟡 高 | APT 可能潜伏 30 天，如何保持状态？内存压力？ |
| **误报控制** | 🟡 高 | 正常扫描器可能也触发多个状态转换 |

**依赖条件**:
```
数据资源:
- 真实 APT 攻击日志 (理想: >= 3 个完整案例)
- 或公开 APT 数据集 (如 MITRE ATT&CK 映射的日志)
- 或威胁情报平台提供的 APT IOC

专业知识:
- APT 分析专家 (至少 1 人)
- 熟悉 MITRE ATT&CK 框架
- 熟悉常见 APT 组织 (APT28, APT29, Lazarus 等)

验证环境:
- 红队演练环境 (模拟 APT 攻击)
- 或购买商业 APT 模拟工具 (如 SafeBreach)
```

**实施风险**:
```
高风险场景:
1. 状态机过于敏感 → 大量误报 (正常扫描被识别为 APT)
2. 状态机过于保守 → 漏报真实 APT
3. 状态无法持久化 → Flink 重启后丢失 APT 追踪
4. 内存消耗过大 → 追踪 1000+ 攻击者的状态

应急预案:
- 简化状态机: 从 6 阶段简化到 3 阶段 (侦察/利用/持久化)
- 降低时间窗口: 从 30 天降到 7 天
- 仅对高分威胁启用状态机 (threatScore > 200)
```

**建议**:
```
✅ 渐进式验证:
   Phase 3 Week 25-26: 实现简化版状态机 (3 阶段)
   Phase 3 Week 27: 使用红队演练数据验证
   Phase 3 Week 28: 根据验证结果调整阈值

✅ 基于规则的状态机 (而非纯 ML):
   - 侦察阶段: uniquePorts > 5 && attackCount > 100
   - 利用阶段: exploitSignature != null (需要 L1 蜜罐数据)
   - 持久化: C&C 连接特征 (需要流量分析)

⚠️ 降级方案:
   - 如果状态机过于复杂，简化为"持续性评分":
     - 仅计算攻击持续时间和频率
     - 不做精细的阶段划分
     - 仍能识别长期威胁
```

## 🟡 High 困难 (技术复杂度高)

### 4. 端口权重科学量化 ⭐⭐⭐⭐

**困难描述**:
- **外部 API 依赖**: 需要调用 NVD (CVSS 评分) 和 AlienVault OTX (威胁情报)
- **API 限流问题**: NVD API 限制 10 req/s，219 个端口需要 22 秒
- **数据更新频率**: CVE 数据每天更新，需要定时同步

**具体挑战**:

| 挑战点 | 难度 | 说明 |
|--------|------|------|
| **NVD API 限流** | 🟡 高 | 10 req/s 限制，批量查询慢 |
| **OTX API Key** | 🟡 高 | 需要注册 AlienVault 账号，可能收费 |
| **数据缓存策略** | 🟡 高 | 如何平衡实时性和 API 调用成本 |
| **离线降级** | 🟡 高 | API 不可用时如何处理？回退到经验权重？ |

**依赖条件**:
```
外部服务:
- NVD API: https://nvd.nist.gov/developers
  - 免费额度: 10 req/s
  - 需要 API Key (免费注册)
- AlienVault OTX API: https://otx.alienvault.com/api
  - 免费额度: 未明确，可能有限制
  - 需要账号注册

网络要求:
- 生产服务器能访问外网 (或通过代理)
- 稳定的网络连接

缓存策略:
- Redis 缓存 CVSS 评分 (7天有效期)
- PostgreSQL 持久化端口权重配置
```

**实施风险**:
```
高风险场景:
1. NVD API 宕机 → 无法更新端口权重
2. API Key 失效 → 服务中断
3. 网络隔离 → 生产环境无法访问外网
4. 成本超预算 → OTX 可能按调用量收费

应急预案:
- 本地缓存 CVSS 数据库 (每周离线更新)
- 回退到经验权重 (219 个端口配置表)
- 混合策略: 高危端口用 CVSS，其他用经验值
```

**建议**:
```
✅ 混合权重策略:
   - 高危端口 (22, 23, 445, 3389 等 50 个) → CVSS API
   - 其他端口 → 经验权重
   - 减少 API 调用到 50 次，5 秒完成

✅ 本地 CVE 数据库:
   - 使用 CVE JSON 数据镜像 (NVD 提供完整下载)
   - 本地 PostgreSQL 存储
   - 每周自动更新

⚠️ 降级方案:
   - 如果外部 API 不可用，直接使用经验权重表
   - 定期人工审核权重 (每季度 1 次)
```

### 5. Flink 状态管理与性能 ⭐⭐⭐⭐

**困难描述**:
- **状态大小爆炸**: 追踪 10000+ 攻击者 × 7 天窗口 = 海量状态
- **Checkpoint 性能**: 状态过大导致 Checkpoint 耗时长，影响吞吐
- **RocksDB 调优**: 需要深入理解 RocksDB 配置参数

**具体挑战**:

| 挑战点 | 难度 | 说明 |
|--------|------|------|
| **状态大小估算** | 🟡 高 | 10000 攻击者 × 7 天 × 1KB/记录 = 70MB/算子 |
| **Checkpoint 时间** | 🟡 高 | 状态 > 1GB 时，Checkpoint 可能超过 1 分钟 |
| **内存 vs 磁盘** | 🟡 高 | RocksDB (磁盘) vs Heap (内存) 性能权衡 |
| **状态清理** | 🟡 高 | TTL 策略设置不当可能导致内存泄漏 |

**依赖条件**:
```
基础设施:
- Flink 集群至少 3 个 TaskManager
- 每个 TaskManager: 8GB 内存, 100GB SSD
- HDFS 或 S3 作为 Checkpoint 存储

Flink 配置:
- state.backend: rocksdb (推荐)
- state.backend.incremental: true
- state.checkpoints.num-retained: 3
- execution.checkpointing.interval: 60s

监控:
- Flink Dashboard 监控状态大小
- Prometheus 监控 Checkpoint 耗时
```

**实施风险**:
```
高风险场景:
1. 状态大小超过磁盘容量 → Checkpoint 失败 → 任务重启
2. Checkpoint 时间过长 → 背压 (Backpressure) → 吞吐下降
3. RocksDB 配置不当 → OOM (Out of Memory)
4. 状态恢复时间长 → 故障恢复慢 (> 5 分钟)

应急预案:
- 启用增量 Checkpoint (减少 I/O)
- 缩短状态 TTL: 7 天 → 3 天
- 分片状态: 按 customerId 分流到多个 Flink Job
```

**建议**:
```
✅ 状态优化:
   - 使用 MapState 而非 ListState (节省空间)
   - 设置合理的 TTL (7 天足够，不需要 30 天)
   - 定期清理过期状态 (每小时触发一次)

✅ 性能测试:
   - Phase 1 Week 15: 10000 events/s 压测
   - 监控状态大小和 Checkpoint 时间
   - 目标: Checkpoint < 30s

⚠️ 降级方案:
   - 如果状态管理性能不足，简化窗口:
     - 取消 7 天演化窗口
     - 只保留 30s / 5min / 15min 窗口
     - 减少状态大小 90%
```

### 6. 多层级时间窗口实现复杂度 ⭐⭐⭐

**困难描述**:
- **Flink 窗口嵌套**: 5 层窗口 (30s / 5min / 15min / 1h / 7d) 需要复杂的流拓扑
- **告警融合逻辑**: 如何合并不同窗口的告警？避免重复告警？
- **性能开销**: 多层窗口导致计算量翻倍

**具体挑战**:

| 挑战点 | 难度 | 说明 |
|--------|------|------|
| **窗口拓扑设计** | 🟡 高 | 5 层窗口需要 5 个独立的 KeyedStream |
| **告警去重** | 🟡 高 | 同一攻击可能在多个窗口触发告警 |
| **延迟 vs 吞吐** | 🟡 高 | 30s 窗口延迟低，7 天窗口吞吐低 |

**建议**:
```
✅ 分阶段实施:
   Phase 1: 实现 3 层窗口 (30s / 5min / 15min)
   Phase 2: 增加 1h 窗口
   Phase 3: 增加 7 天窗口

✅ 告警融合:
   - 使用 CoProcessFunction 合并多层窗口结果
   - 优先级: 30s > 5min > 15min (快速窗口优先)

⚠️ 降级方案:
   - 如果性能不足，简化为 2 层窗口:
     - 快速窗口: 30s (勒索软件)
     - 主窗口: 15min (通用威胁)
```
